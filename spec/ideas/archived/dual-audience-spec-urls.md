## Problem Statement

How might we make the specscore.md specification site equally first-class for AI agents (who benefit from raw Markdown) and humans (who benefit from navigable HTML), without changing the canonical adherence-footer URL convention or inventing content-negotiation infrastructure — just by productizing the dual-output pipeline that already exists?

## Context

The static site generator at `tools/site-generator/` already produces **both** an HTML and a Markdown version of every specification page and places them side by side in `public/`. Firebase Hosting with `cleanUrls: true` serves:

- `https://specscore.md/feature-specification` → `feature-specification.html` (rich HTML site with sidebar nav, branding, theme toggle)
- `https://specscore.md/feature-specification.md` → raw Markdown with links rewritten to `.md` siblings so agent navigation is self-contained

The infrastructure is solid. What is missing is **discoverability and documentation** of the dual-URL pattern. Consequences:

- The HTML `<head>` does not carry a `<link rel="alternate" type="text/markdown">` pointer to the `.md` sibling. Well-behaved agent tools and crawlers that look for alternate representations find nothing.
- There is no `/llms.txt` or `/llms-full.txt`, the emerging convention (adopted by Anthropic docs, OpenAI docs, Cursor, Mintlify, and growing) for giving agents a curated, low-token index of a site's important content.
- No page on the site tells agent maintainers or curious users that the `.md` suffix works.
- HTTP response headers do not advertise the alternate representation (`Link: </slug.md>; rel="alternate"; type="text/markdown"`).
- The domain name `specscore.md` hints at Markdown-first, but a first-time visitor who lands on a spec URL gets HTML. That is the right choice for first impressions, but the `.md` hint never gets delivered unless the user already knew to append `.md`.

This Idea also reopens — briefly — the question of whether the canonical adherence-footer URL should remain HTML-default, flip to MD-default, or serve both via content negotiation. The Recommended Direction takes a position; Alternatives Considered documents the cases that lost and why.

## Recommended Direction

Keep HTML as the default at the canonical URL (`https://specscore.md/{slug}`). The adherence-footer URL convention stays as specified in the `adherence-footer-and-doc-type-registry` Idea. Invest in making the Markdown sibling **maximally discoverable** to agents without degrading the human first-impression or link-preview experience.

Five concrete productization moves:

1. **`<link rel="alternate" type="text/markdown" href="/{slug}.md">`** in every generated HTML `<head>`. Web-standard. Zero cost. Respected by well-behaved crawlers and agent tools.
2. **Visible "View as Markdown" link** in the HTML sidebar or page header — the same affordance GitHub gives with its "Raw" button. Human- and agent-discoverable.
3. **`/llms.txt`** (curated index, following the Jeremy Howard convention) and **`/llms-full.txt`** (concatenated Markdown corpus of all spec pages + key onboarding docs). Generated by the site-generator alongside existing outputs.
4. **HTTP `Link` response header** emitted by Firebase Hosting: `Link: </slug.md>; rel="alternate"; type="text/markdown"`. Advertised at the network layer so agents that never parse HTML still discover the alternate.
5. **New page `/for/ai-agents`** explaining the conventions: `.md` suffix, `llms.txt`, `llms-full.txt`, `<link rel="alternate">`, and a short "how to configure your agent to use SpecScore" section. Slot already exists — `/for/developers`, `/for/product-owners`, etc.

The upstream `adherence-footer-and-doc-type-registry` Idea does not change as a result of this Idea. Footer URLs remain bare HTML-default URLs. Agent discoverability is a separate concern handled at the publishing layer, not at the authored-spec layer.

## Alternatives Considered

**Flip the canonical URL to MD-default** — `https://specscore.md/{slug}` returns raw Markdown with a prominent link at the top to the HTML-rendered version. The `.md` domain hint is fulfilled literally. Rejected for five reasons:

1. First-time human visitors (from Google results, Twitter shares, marketing links) land on raw text with no navigation. Unpolished first impression.
2. Link previews (Slack, iMessage, Discord, Twitter) depend on `<meta og:*>` tags that exist only in HTML. MD-default makes shared links preview blank.
3. SEO signals (title tags, structured data, headings parsed as hierarchy) are weaker for raw MD; search rankings suffer.
4. Every page becomes a dead end — raw MD has no sidebar, so the returning-developer experience ("skim spec → jump to related spec via sidebar") breaks.
5. The only scenario MD-default cleanly wins is "agent that isn't configured to check alternate representations and hasn't fetched `llms.txt`" — a shrinking set as `<link rel="alternate">` and `llms.txt` adoption grows. HTML-default's wins (first-impression, social, SEO) are permanent.

**Content negotiation via Firebase Functions** — route `/{slug}` through a Cloud Function that returns HTML when `Accept: text/html` dominates and MD when `Accept: text/markdown` dominates. One canonical URL for both audiences. Rejected because it adds a Cloud Function to a site that is otherwise fully static, complicates caching and CDN behavior, makes local-dev and debugging harder, and closes a discoverability gap that `<link rel="alternate">` + `llms.txt` already close for almost the same agent population at a fraction of the operational cost.

**Put the `.md` sibling URL directly in the adherence footer** (the alternative that was rejected during sharpening question 2 in the ideation session). Rejected because the `REQ: adherence-footer` rule explicitly requires a single bare URL, chosen so footers stay brand-neutral and minimal. Adding a second URL or a Markdown-link wrapper breaks that. Discoverability should live at the publishing layer, not inside every authored document.

**Do nothing — status quo** — accept that agents who want MD must already know to append `.md`. Rejected because the cost of the five moves above is low enough that doing nothing is strictly worse than doing them. Agents being opaque about how to find raw content is a form of friction SpecScore can eliminate for free.

## MVP Scope

One cycle, scoped to the site-generator and site content. Five concrete deliverables, each independently shippable and revertible:

1. `<link rel="alternate">` in the HTML template
2. "View as Markdown" visible link in the HTML shell
3. `llms.txt` and `llms-full.txt` generation wired into the build pipeline, driven by the existing `site-config.json`
4. Firebase `headers` configuration emitting the `Link: </slug.md>; rel="alternate"` header for HTML responses
5. `/for/ai-agents` content page explaining all of the above

No changes to authored specs, no changes to the adherence-footer rule, no backend services, no content negotiation.

## Not Doing (and Why)

- **Flipping the canonical URL to MD-default** — rejected above; would require a repo-wide rewrite of adherence-footer URLs in all authored specs and degrade human-facing UX.
- **Content negotiation via Cloud Functions** — rejected above; introduces compute to a static site for marginal agent-side benefit.
- **Putting multiple URLs in each adherence footer** — violates the single-bare-URL REQ. Discoverability belongs at the publishing layer, not the authoring layer.
- **Changing the adherence-footer-and-doc-type-registry Idea** — this Idea is strictly additive to the publishing pipeline. The footer URL convention is unchanged.
- **Building a custom agent-discovery protocol** — `<link rel="alternate">` and `llms.txt` are existing conventions. Do not invent new ones.
- **Generating per-spec `llms.txt` files** (one per directory) — `llms.txt` is a site-level convention. One file at the root suffices.
- **Integrating server-side analytics to measure "agent vs human" traffic** — interesting but not required to ship this. Revisit post-launch if we want to measure discoverability impact.

## SpecScore Integration

- **New Features this would create:** None in `spec/features/`. This Idea lives at the publishing/infrastructure layer; it does not introduce SpecScore document types.
- **Existing Features affected:** None in the normative spec. `adherence-footer-and-doc-type-registry` is strictly upstream; this Idea honors its convention rather than modifying it.
- **Site-generator changes:** `tools/site-generator/` gains a step to emit `llms.txt` and `llms-full.txt`; HTML template gains `<link rel="alternate">` and a visible MD link; `site-config.json` may gain an optional per-page `llmsDescription` field for the curated index.
- **Firebase config changes:** `firebase.json` gains a `headers` block emitting the `Link` header for HTML responses.
- **Content additions:** `/for/ai-agents` page under `docs/for/ai-agents.md` (or similar), registered in `site-config.json`.
- **Dependencies:** Depends on `adherence-footer-and-doc-type-registry`. That Idea defines the adherence-footer URL shape this Idea honors; it also defines the Document/Index Kind taxonomy that informs which pages belong in `llms.txt` (all Document and Index Kind entries; Meta and Structure entries get de-prioritized or omitted).

## Open Questions

- **llms.txt scope** — should `llms-full.txt` include only specification pages, or also onboarding (`/install`, `/demo/todo-app`) and role-based guides (`/for/developers`, etc.)? Current lean: specs + install + demo. Role-based guides are situational and arguably noisy for an agent bootstrapping SpecScore context.
- **"View as Markdown" placement** — sidebar section, below the page title, a kebab menu on the page header, or multiple? Design decision at implementation time.
- **llms.txt description content** — is the one-line description per page generated from the first sentence of the spec's Summary section, or hand-curated per entry in `site-config.json`? Current lean: hand-curated for the small specifications set (7–10 entries); automatic for any future larger corpus.
- **Cache-Control headers for `.md` responses** — should raw `.md` be cached differently than HTML given typical consumer (agent) patterns (frequent, short-lived, varied origins)? Current lean: same cache policy as HTML; revisit if agent fetch patterns warrant tuning.
- **Versioning `llms-full.txt`** — should the file carry a `Last-Modified` or content hash footer so agents can detect changes and invalidate their cached context? Current lean: rely on HTTP `ETag` / `Last-Modified`; do not invent a file-level versioning scheme.
- **Measurement** — do we add request analytics differentiating `.md` vs. `.html` traffic to quantify agent adoption post-launch? If so, what platform (Firebase Analytics, Cloudflare, Plausible)? Out of scope for the MVP; flagged for a later Idea if the signal is wanted.
